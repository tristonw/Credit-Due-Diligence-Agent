## 一、架构总览

### 1.1 定位

> **不是做重型 Workflow Engine，而是做：面向尽调报告生产的 Task Manifest 轻量级编排系统。**

### 1.2 核心设计哲学

| 概念 | 说明 |
|------|------|
| **Manifest** | 运行时单一真相源，不是静态备注文件 |
| **dd_coordinator** | 负责推进主阶段，不再承担重型 workflow controller 角色 |
| **dd_chapter_writer** | 不只是起子任务，而是要把章节 item 真正闭环完成 |
| **Trick Layer** | 业务 know-how 的系统化注入层，不依赖 Agent 自己想起来加载 |
| **dd_expert_agent** | 不主导单次运行，而是从历史和反馈中推动系统持续进化 |

### 1.3 总体架构层次

```
┌─────────────────────────────────────────┐
│  第4层：用户交互层                        │
│  - 用户请求入口                           │
│  - 最终报告输出                           │
└─────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────┐
│  第3层：协调控制层 (dd_coordinator)       │
│  - 主阶段推进                             │
│  - 状态管理                               │
│  - 轻量级协调（非重型workflow）            │
└─────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────┐
│  第2层：章节执行层 (dd_chapter_writer)    │
│  - 章节内容生成                           │

│  - Item级别完成                           │
└─────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────┐
│  第1层：专家智能体层 (dd_expert_agent)    │
│  - 领域专家知识                           │
```

## 二、核心概念

### 2.1 为什么需要新方案

**传统多Agent方案的问题：**
- ❌ Agent执行完即结束，没有质量兜底
- ❌ 失败即整轮重跑，效率低下
- ❌ 反馈脱离系统，无法持续改进
- ❌ 依赖Agent记忆，知识无法固化
- ❌ 单次运行，无法持续进化

**AForce Manifest方案的改进：**
- ✅ Completion Check + Trick 质量有保障
- ✅ 局部修复/重试 效率高
- ✅ 反馈回流为修订任务 持续改进
- ✅ Trick Layer系统化注入 知识固化
- ✅ Expert Agent持续进化 越用越聪明

### 2.2 三层平台能力

> **新的尽调 Manifest 方案，不是单纯把"多 Agent"换个名字，而是补上了三层真正的平台能力：**

| 层级 | 能力 | 价值 |
|------|------|------|
| **Manifest** | 把报告生成过程变成**可见、可控、可干预**的任务清单 | 运行时单一真相源 |
| **Trick** | 把尽调 know-how **工程化**为系统级**强制注入**能力 | 质量兜底 |
| **Expert Agent** | 把失败案例和用户反馈变成**可持续迭代**的闭环 | 持续进化 |

## 三、四层架构

### 3.1 第一层：用户交互层

- **职责**：用户请求入口、最终报告输出
- **关键**：用户体验、报告展示

### 3.2 第二层：协调控制层 (dd_coordinator)

- **职责**：主阶段推进、状态管理
- **设计**：轻量级协调，非重型workflow controller
- **关键**：推进而非控制，状态管理而非流程编排

### 3.3 第三层：章节执行层 (dd_chapter_writer)

- **职责**：章节内容生成、质量闭环
- **设计**：不只是起子任务，而是要把章节 item 真正闭环完成
- **关键**：完成确认、质量校验、自动修复

### 3.4 第四层：专家智能体层 (dd_expert_agent)

- **职责**：领域专家知识、历史经验学习
- **设计**：不主导单次运行，而是从历史和反馈中推动系统持续进化
- **关键**：模式识别、根因分析、优化建议

### 3.5 Trick Layer (贯穿各层)

- **职责**：业务 know-how 的系统化注入
- **设计**：不依赖 Agent 自己想起来加载
- **关键**：规则库、模板、校验器、最佳实践

## 四、运行时主闭环

### 4.1 问题定义

> **一次尽调报告从"发起生成"到"拿到最终报告"的运行时闭环是怎样的？**

### 4.2 运行流程

```
用户请求 → 生成Manifest → 阶段推进 → 章节执行 → 质量检查 → 专家审核 → 报告输出
       ↑________________________________________________↓
                          反馈循环
```

### 4.3 关键设计

| 关键点 | 说明 | 设计意图 |
|--------|------|----------|
| **Completion Check + Trick** | 每个阶段都要经过"完成检查+技巧验证" | 不只是Agent执行完就算了，必须有质量把关 |
| **局部修复/重试** | 阻塞后不要求整轮重跑，可回到Manifest做局部修复 | 提高容错性，降低重试成本 |
| **反馈回流** | 用户反馈不会脱离系统，而是回流为新的修订任务 | 形成持续改进闭环 |

## 五、章节级闭环

### 5.1 问题定义

> **为什么新的方案不只是"任务可见了"，而是真的能把报告质量一步步兜住。**

### 5.2 核心设计

| 设计点 | 说明 | 工程价值 |
|--------|------|----------|
| **系统不信任"自报完成"** | Agent生成内容后，必须经过校验 | 防止Agent幻觉/敷衍 |
| **completed是校验和修复后得出的** | 章节item的完成状态是验证后的结果 | 质量有保障 |
| **advisor保留业务警告但不阻断** | 风险提示与流程推进分离 | 灵活性与严谨性平衡 |
| **不靠单次prompt运气** | 靠章节级闭环逐步保证效果 | 可预期的稳定性 |

### 5.3 校验器说明

| 校验器 | 职责 | 修复方式 |
|--------|------|----------|
| **file_format_validator** | 格式规范性检查 | auto_fix_transformer |
| **content_completeness_validator** | 内容完整性检查 | auto_fix_transformer |
| **domain_validator** | 领域专业性检查 | advisor + 人工介入 |

## 六、Trick自闭环

### 6.1 问题定义

> **Trick 不只是一次运行时的检查器，而是如何自己形成"发现问题 -> 沉淀规则 -> 下次更稳"的闭环。**

### 6.2 核心含义

| 核心点 | 说明 |
|--------|------|
| **问题不只是"修完就算"** | 当前运行暴露的问题要结构化记录 |
| **沉淀为结构化运行记录** | 问题要转化为可查询、可分析的数据 |
| **专家Agent分析高频失败点** | 从数据中发现模式和根因 |
| **产出改进建议** | 对Trick库、模板规范、默认任务清单的优化 |
| **人工审核后进入下一循环** | 人的判断确保改进方向正确 |

### 6.3 自闭环流程

```
当前运行 → 问题结构化记录 → 运行记录库 → Expert Agent分析 → 改进建议 → 人工审核 → 下一循环
```

## 七、业务闭环

### 7.1 问题定义

> **用户为什么会感受到报告越来越稳、越来越像"尽调产品"而不是"通用大模型输出"。**

### 7.2 业务闭环要表达什么

| 核心观点 | 说明 |
|---------|------|
| **一次报告不是终点** | 用户反馈是下一次运行的输入 |
| **平台沉淀的是"尽调经验"** | 不只是沉淀聊天记录，而是可复用的经验 |
| **多Agent只是执行层** | 真正形成差异化的是 **Manifest + Trick + Expert Agent** 的组合 |

## 八、两条核心闭环

### 8.1 闭环1：运行时质量闭环

```
Agent → Trick → Completion → Report
```

**说明**：执行 → 校验 → 完成确认 → 交付

### 8.2 闭环2：持续优化闭环

```
Report Feedback → Expert Agent → Trick Library → Next Run
```

**说明**：反馈 → 分析 → 改进 → 下次更稳

## 九、对外汇报话术

### 9.1 开场定位（30秒）

> "我们不是在做重型 Workflow Engine，而是在做**面向尽调报告生产的 Task Manifest 轻量级编排系统**。**"

### 9.2 运行闭环（1分钟）

> "用户发起任务后，系统会：
> 1. **Manifest** 把任务拆成阶段和章节 Item
> 2. **多 Agent** 分工执行
> 3. 每个阶段都要经过 **Trick** 校验和完成确认
> 4. 失败后可人工介入和局部重跑
> 5. 最终得到报告产物"

### 9.3 持续优化（1分钟）

> "更重要的是，一次报告不是终点：
> - 每次运行都会**沉淀问题**与处理方式
> - **Expert Agent** 分析高频失败点
> - 沉淀为新的 **Trick、规范和默认任务**
> - 下一次同类报告**更稳、更快、更像业务专家产物**"

### 9.4 核心价值（30秒）

> "这套方案补上了三层真正的平台能力：
> - **Manifest**：可见、可控、可干预的任务清单
> - **Trick**：系统级强制注入的 know-how
> - **Expert Agent**：可持续迭代的优化闭环
> 
> 最终形成**运行时质量闭环**和**持续优化闭环**两条核心循环。**"

## 十、结论

### 10.1 核心观点

> **新的尽调 Manifest 方案，不是单纯把"多 Agent"换个名字，而是补上了三层真正的平台能力：**

| 层级 | 能力 | 价值 |
|------|------|------|
| **Manifest** | 把报告生成过程变成**可见、可控、可干预**的任务清单 | 运行时单一真相源 |
| **Trick** | 把尽调 know-how **工程化**为系统级**强制注入**能力 | 质量兜底 |
| **Expert Agent** | 把失败案例和用户反馈变成**可持续迭代**的闭环 | 持续进化 |

### 10.2 核心价值

- ✅ **系统能力随运行次数增长**，不是永远靠某个人临场经验兜底
- ✅ **报告越来越稳、越来越像"尽调产品"**而非"通用大模型输出"
- ✅ **真正形成差异化的是 Manifest + Trick + Expert Agent 的组合**

### 10.3 两条核心闭环总结

**闭环1：运行时质量闭环**
```
Agent → Trick → Completion → Report
```

**闭环2：持续优化闭环**
```
Report Feedback → Expert Agent → Trick Library → Next Run
```
